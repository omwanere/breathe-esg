# Architectural Decisions Document (ADD)

This document explains the technical choices, calculations, and assumptions made during the implementation of the Breathe ESG platform.

---

## 1. Pipeline & Ingestion Mode Decisions

### SAP Ingestion: Flat File over IDoc or OData RFC
* **Decision**: Implement CSV/XLSX MB51 flat-file exports.
* **Rationale**: SAP ERP setups typically reside behind strict corporate firewalls. Direct RFC/OData connection configuration requires complex network permissions (VPNs, SAP Router setups) and months of enterprise security approval. In contrast, transaction MB51 (Material Movements) is accessible to accounting/operations users out-of-the-box. CSV exports are easy to schedule and upload immediately.

### SAP Movement Types Filter
* **Decision**: Filter rows to movements `261` (goods issue to production), `201` (goods issue to cost center), and `101` (goods receipt for order/production).
* **Rationale**: Movement type `261` and `201` capture direct consumption of fuel/materials drawn from inventories. Movement type `101` captures purchased receipts, representing upstream procurement. Other movement types (e.g., `311` transfer postings between plants, `561` initial stock entries) represent inventory shifting, not active fuel burning, and are excluded to prevent double counting.

### Utility Ingestion: Portal CSV over PDF OCR or API
* **Decision**: Portal CSV upload.
* **Rationale**: Utility APIs (such as Green Button) lack standardization across global vendors, and PDF OCR engines are error-prone and sensitive to format adjustments. Direct CSV exports from EDF, PG&E, or business billing portals are clean, machine-readable, and easily obtainable by administrative staff.

### Travel Ingestion: Concur CSV Report over Live OAuth API
* **Decision**: CSV travel report.
* **Rationale**: Establishing corporate Concur API access requires Enterprise Admin OAuth setup, which client security departments block during initial ESG onboarding. Compliance analysts typically receive travel summaries as CSV files generated by internal finance desks.

## 2. Calculation & Normalization Decisions

### IATA Flight Distance: Great-Circle (Haversine) vs Actual Path
* **Decision**: Great-Circle distance calculated via the Haversine formula, using a bundled airport coordinates map.
* **Rationale**: Airline flight trajectories change based on weather patterns, air traffic control routing, and holdings. Great-Circle distance represents the standardized baseline distance mandated by DEFRA and the GHG Protocol. It is calculated by extracting decimal latitudes and longitudes for airport pairs and applying:
  $$d = 2 R \arcsin\left(\sqrt{\sin^2\left(\frac{\Delta \phi}{2}\right) + \cos(\phi_1)\cos(\phi_2)\sin^2\left(\frac{\Delta \lambda}{2}\right)}\right)$$
  Where $R = 6371.0\text{ km}$.

### Flight Cabin Class Multipliers
* **Decision**: Economy (1.0x), Premium Economy (1.6x), Business (2.0x), First (2.4x).
* **Rationale**: First-class and business-class seats occupy larger physical areas (footprint) on an aircraft, reducing overall passenger capacity and increasing the carbon share per passenger. We follow the DEFRA guidelines to apply these multipliers directly to computed flight distances.

### Default Emission Factor Selection
* **Decision**: DEFRA (UK Department for Environment, Food & Rural Affairs) 2023 is used as the global default, alongside EPA eGRID 2022 for US electricity grids.
* **Rationale**: DEFRA publishes one of the most comprehensive, standardized, and publicly audited lists of carbon factors globally, covering transport, materials, and hotel room-nights. EPA eGRID is the standard source for US electricity grid location-based factors.

### Billing Period Crossing Month Boundaries
* **Decision**: Allocate the consumption date to the midpoint of the billing period.
* **Rationale**: Utility invoices often cover dates like Dec 14 to Jan 13. Midpoint allocation is standard and ensures the activity date falls squarely in the month representing the majority of the consumption cycle.

## 3. Compliance & Locking Decisions

### Multi-Tenancy Strategy
* **Decision**: Row-level Tenant FK isolation.
* **Rationale**: Facilitates immediate multi-tenant deployment in SQLite/Postgres without complex connection pool switching, simplifies migrations, and keeps operational infrastructure costs low.

### Audit Lock Criteria
* **Decision**: Rows are marked `is_locked = True` once they are exported for audit.
* **Rationale**: Locking prevents historical data manipulation after reports have been reviewed, approved, and delivered to auditors. Locked rows block PATCH and POST actions.

## 4. Product Questions for the PM

To align requirements, we would clarify the following questions with the Product Manager:
1. **SAP Plant Resolution**: Are SAP plant code translations (Werk -> Location) maintained centrally by Breathe ESG admins, or does each tenant configure their own mappings?
2. **Reviewer Limits**: Can analysts edit quantities and units directly in staging, or should edits require approval from a senior manager?
3. **Multi-Currency Travel**: Concur logs transactions in local currencies. Should the backend call a live conversion API (e.g. OpenExchangeRates) to convert to USD, or assume the input CSV has pre-converted columns?
4. **Utility Scope Mappings**: Should we support Scope 2 (Location-Based) and Scope 2 (Market-Based) grid factors simultaneously using electricity contracts?
5. **Re-upload Policy**: If a file contains rows that have already been ingested, should we overwrite, skip, or create duplicate entries?
